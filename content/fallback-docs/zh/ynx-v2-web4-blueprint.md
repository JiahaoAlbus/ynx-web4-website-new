# YNX v2 Web4 蓝图（中文）

状态：active draft  
最后更新：2026-02-25

## 1. 定位

YNX v2 的目标不是“普通 EVM 链微调”，而是：

- 维持以太坊开发体验（EVM 工具链兼容）
- 追求高性能低延迟（接近 Solana 类体感）
- 面向 AI 与 Web4 场景的去中心化结算层

一句话：**AI-native Web4 链，开发体验像 ETH，性能目标向高性能链看齐。**

## 2. 核心目标

- 体感确认目标：正常网络下 ≤ 1s
- 最终确认目标：区域内 2–4s，跨洲 ≤ 6s
- 完全链上治理：参数、资金、升级路径可审计
- 验证人开放加入：持续去中心化
- AI 任务可结算：任务登记、结果提交、挑战仲裁、奖惩结算

## 3. 架构方向

### 3.1 执行层

- 保留 EVM 兼容
- 引入并行执行路线（冲突检测 + 多执行通道）

### 3.2 共识层

- 保留 BFT 快终局
- 使用可切换参数档位：
  - 快速档（低延迟区域）
  - 稳定档（跨洲优先）

### 3.3 费用层

- 支持 base fee + priority fee
- 引入本地费市场思路，降低全局拥堵耦合
- 面向开发者支持赞助 Gas / AA 路线

### 3.4 AI 结算层

- 任务创建（需求方）
- 节点执行与结果承诺（供给方）
- 挑战窗口与争议处理
- 奖励与惩罚结算

## 4. 与 v1 的关系

- v2 是新轨道，不直接在 v1 公测网上硬改
- 使用新 chain-id + 新 genesis
- 先完成 v2 公测网，再进入主网决策

## 5. 当前仓库交付定义

本仓库内“v2 完整轨道”指：

- 有明确 v2 协议规范
- 有可执行的 v2 启动与配置脚本
- 有可观测的 v2 对外元数据接口
- 有可复制的运营/接入文档

主网上线仍需在公测网通过性能与稳定性验收后推进。
